# 专家分析框架参考

## 调度专家工作流模板

### 问题分析清单
```
□ 问题的核心是什么？
□ 涉及哪些利益相关者？
□ 有哪些约束条件（时间、预算、资源）？
□ 问题的紧急程度如何？
□ 需要哪些领域的专业知识？
□ 各维度之间是否存在依赖关系？
```

### 工作流设计模板
```markdown
## 问题：[一句话描述]

### 复杂度评估
- 简单（1-2个维度）
- 中等（3-4个维度）
- 复杂（5+个维度）

### 涉及维度
1. [维度1] - 关联专家：[专家名]
2. [维度2] - 关联专家：[专家名]
...

### 执行策略
- 并行：各专家独立分析，最后汇总
- 串行：专家按依赖顺序依次分析
- 混合：部分并行，部分串行

### 专家任务分配
| 专家 | 核心任务 | 输入依赖 | 输出要求 |
|-----|---------|---------|---------|
| 专家A | ... | 无 | ... |
| 专家B | ... | 专家A输出 | ... |
```

## 技术架构专家分析框架

### 技术评估维度
1. **功能性**：是否满足核心需求
2. **性能**：吞吐量、延迟、并发能力
3. **可靠性**：可用性、容错性、数据一致性
4. **安全性**：认证授权、数据保护、合规性
5. **可维护性**：代码质量、文档、社区活跃度
6. **可扩展性**：水平/垂直扩展能力
7. **集成性**：与现有系统的兼容度

### 技术选型矩阵模板

## 汇总专家整合框架

### 观点整合流程
1. **收集**：汇总所有专家的核心观点
2. **分类**：按主题或维度归类
3. **对比**：识别共识和分歧
4. **权衡**：评估不同观点的权重
5. **综合**：形成统一的解决方案

### 冲突解决策略
```markdown
## 观点冲突记录
| 冲突点 | 专家A观点 | 专家B观点 | 冲突原因 | 解决方案 |
|-------|----------|----------|---------|---------|
| 冲突1 | ... | ... | ... | ... |

## 决策原则
1. [原则1] - 适用场景：...
2. [原则2] - 适用场景：...
```

### 方案评估清单
```
□ 方案是否覆盖了问题的所有关键维度？
□ 各专家的核心建议是否都被合理整合？
□ 方案是否具有可执行性？
□ 是否考虑了约束条件？
□ 风险是否被充分识别和缓解？
□ 成功指标是否清晰可衡量？
□ 方案是否易于理解和沟通？
```

## 常见场景专家组合建议

### 技术决策场景
- **核心专家**：技术架构专家、量化思维专家
- **辅助专家**：系统分析专家（涉及组织变革时）

### 战略规划场景
- **核心专家**：系统分析专家、成长规划专家
- **辅助专家**：量化思维专家（需要财务分析时）

### 个人发展场景
- **核心专家**：成长规划专家、系统分析专家
- **辅助专家**：量化思维专家（评估投资回报时）

### 产品创新场景
- **核心专家**：系统分析专家、技术架构专家、成长规划专家
- **辅助专家**：量化思维专家

### 组织变革场景
- **核心专家**：系统分析专家、成长规划专家
- **辅助专家**：技术架构专家（涉及系统改造时）、量化思维专家
