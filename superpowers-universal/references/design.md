# Phase 2: DESIGN - Create Approach

## Overview

**Goal:** Decide HOW to achieve the clarified intent.

**When to Use:** After CLARIFY phase is complete and intent is understood.

**Success Criteria:** User approves a specific approach.

## The Process

### Step 1: Generate Alternatives

**Rule:** Always present 2-3 different approaches. Never present just one option.

**Why multiple options?**
- Forces you to think creatively
- Gives user agency in decision
- Surfaces trade-offs explicitly
- Reduces confirmation bias

### Step 2: Present Trade-offs

For each alternative, clearly state:
- **Pros:** What's good about this approach
- **Cons:** What's challenging or risky
- **Best for:** When this approach shines

### Step 3: Make a Recommendation

**Always recommend one option** with clear reasoning:
- "I recommend Option B because..."
- Connect recommendation to user's specific context
- Show you understand their priorities

### Step 4: Present Design in Sections

**Scale sections to complexity:**
- Simple task: A few sentences per section
- Complex task: Up to 200-300 words per section

**Sections to Cover:**

#### 1. Approach (High-Level Strategy)
- What's the overall method?
- Why this approach fits the goal?

#### 2. Structure (Components/Parts)
- What are the main pieces?
- How do they fit together?

#### 3. Flow (Sequence/Organization)
- What order will things happen?
- Are there dependencies?

#### 4. Key Decisions
- What critical choices were made?
- Why were they made?

#### 5. Risk Handling
- What could go wrong?
- How will we mitigate?

### Step 5: Get Incremental Validation

**Present one section at a time** for complex designs:

```
"Let me walk you through the design section by section.

**APPROACH:** [Description]

Does this high-level approach feel right?"
```

**Wait for approval before continuing to next section.**

## Design Patterns

### Pattern 1: The Comparison Table

For clear side-by-side comparison:

```
| Approach | Pros | Cons | Best For |
|----------|------|------|----------|
| A: Quick | Fast, simple | Less thorough | Time-critical |
| B: Thorough | Comprehensive | Takes longer | High stakes |
| C: Hybrid | Balanced | Moderate complexity | Most cases |
```

### Pattern 2: The Narrative Comparison

For more nuanced discussion:

```
**Option A: [Name]**
[Description of approach]

This works well because [pros]. The trade-off is [cons].

**Option B: [Name]**
[Description of approach]

This takes longer but delivers [pros]. The challenge is [cons].

**My Recommendation: Option [X]**
Given your [specific context], I recommend Option X because [reasoning].
```

### Pattern 3: The Visual Outline

For structural clarity:

```
**Proposed Structure:**

1. [Section A]
   - [Subsection]
   - [Subsection]

2. [Section B]
   - [Subsection]
   - [Subsection]

3. [Section C]
   - [Subsection]
```

## Anti-Pattern: "This Is Too Simple To Need A Design"

**Reality Check:**

Every task benefits from design, even if brief:

| Task | Brief Design |
|------|-------------|
| Short email | "Direct approach: state purpose, provide context, clear ask" |
| Simple checklist | "Organize by priority: must-do, should-do, nice-to-do" |
| Quick analysis | "Top-down: summary first, then supporting details" |

**The 30-Second Design:**
Even a simple sentence describing approach counts as design.

## Examples

### Example 1: Writing Task

**Context:** Email to client about project delay

**Design:**

```
**Option A: Direct and Brief**
- State delay upfront
- Brief technical explanation
- New timeline
- **Pros:** Respects their time, no fluff
- **Cons:** May seem abrupt
- **Best for:** Busy executives, established relationships

**Option B: Relationship-Focused**
- Acknowledge partnership first
- Frame as quality investment
- Context before news
- **Pros:** Maintains trust, shows care
- **Cons:** Longer read
- **Best for:** New relationships, sensitive situations

**My Recommendation: Option B**
Given that this is an important client relationship and the delay is significant, maintaining trust is as important as the timeline. The relationship-focused approach positions the delay as investing in quality rather than missing deadlines.

**Design Details:**

**Approach:** Relationship-first communication

**Structure:**
1. Opening: Acknowledge partnership
2. The News: Clear delay statement
3. Context: Why (brief, non-technical)
4. Our Response: What we're doing
5. Ask: Extension request
6. Closing: Quality commitment

**Flow:** Warm → Factual → Action-oriented

**Key Decisions:**
- Lead with relationship, not delay
- Keep technical details minimal
- Position as quality investment

Does this approach feel right?
```

### Example 2: Analysis Task

**Context:** Sales dropped 20%, need to find causes

**Design:**

```
**Option A: Data-First Deep Dive**
- Comprehensive data analysis
- Pattern identification
- Hypothesis testing
- **Pros:** Thorough, objective
- **Cons:** Time-intensive (2-3 days)
- **Best for:** When cause is completely unknown

**Option B: Hypothesis-Driven**
- Start with your hypotheses
- Test top 2-3 quickly
- Expand if needed
- **Pros:** Fast if hypotheses are right (4-6 hours)
- **Cons:** May miss other causes
- **Best for:** When you have strong initial hypotheses

**Option C: Mixed Approach** ⭐ RECOMMENDED
- Quick data scan for obvious patterns (1 hour)
- Test your top 2 hypotheses (2 hours)
- Deep dive if neither explains it (remaining time)
- **Pros:** Balanced speed and thoroughness
- **Cons:** Moderate time investment
- **Best for:** Most business analysis situations

**My Recommendation: Option C**
This gives us the best of both worlds - we'll quickly validate or rule out your hypotheses, but also keep an eye open for patterns you might not have considered.

**Design Details:**

**Approach:** Iterative hypothesis testing with open discovery

**Structure:**
1. Data extraction and quick scan
2. Hypothesis A testing
3. Hypothesis B testing
4. Alternative cause exploration (if needed)
5. Root cause ranking and validation

**Flow:** Broad scan → Focused testing → Deep dive (if needed)

**Key Decisions:**
- Test hypotheses first (efficient if right)
- Keep alternative exploration ready
- Rank causes by impact and evidence

Sound good?
```

### Example 3: Planning Task

**Context:** 2-day team offsite for 15 people

**Design:**

```
**Option A: Conference Center**
- Professional meeting rooms
- Built-in AV and catering
- Standard offsite format
- **Pros:** Focused environment, reliable
- **Cons:** Less character, can feel corporate
- **Best for:** Strategy-heavy, formal teams

**Option B: Boutique Hotel** ⭐ RECOMMENDED
- Meeting space + character
- On-site dining
- More relaxed atmosphere
- **Pros:** Good for team building + work
- **Cons:** May have AV limitations
- **Best for:** Balancing work and connection

**Option C: Retreat Center**
- Nature setting
- Purpose-built for offsites
- Activities available
- **Pros:** Best for bonding
- **Cons:** May feel too casual for serious strategy
- **Best for:** Team building primary goal

**My Recommendation: Option B - Boutique Hotel**
Given your 70% strategy / 30% team building split, the boutique hotel provides professional space for strategic work while the atmosphere supports relationship building.

**Design Details:**

**Approach:** Boutique hotel with structured agenda

**Structure:**

*Day 1:*
- Morning: Travel + casual lunch (team building)
- Afternoon: Strategic session 1 (planning)
- Evening: Group dinner + fun activity (team building)

*Day 2:*
- Morning: Strategic session 2 (planning)
- Afternoon: Action planning + travel home

**Flow:** Ease in → Intensive work → Connect → Intensive work → Wrap up

**Key Decisions:**
- Start with casual connection (lower pressure)
- Front-load strategy work when energy is high
- End with action planning (clear next steps)

Does this framework work?
```

## Handling User Feedback

### If User Says "Yes, proceed"

```
"Great! Moving to PLAN phase to break this into specific steps."
```

### If User Says "Not quite..."

```
"I understand. Let me revise.

[Adjust based on feedback]

How about now?"
```

### If User Has a Different Idea

```
"That's a great point. Let me incorporate that...

[Revise with their input]

Does this adjusted approach capture what you're thinking?"
```

## Transition to PLAN

**When to Move On:**
- User explicitly approves the approach
- All sections validated (for complex designs)
- You have clear direction

**How to Transition:**
```
"Perfect, we have an approved approach.

**Approved Design:** [Brief summary]

Next, I'll create a detailed plan breaking this into specific, actionable steps. I'll include what needs to be done and how we'll verify each step is complete. Ready?"
```

## Quick Checklist

- [ ] Generated 2-3 distinct alternatives
- [ ] Clearly stated pros/cons for each
- [ ] Made explicit recommendation with reasoning
- [ ] Presented design in appropriate sections
- [ ] Got user approval on approach
- [ ] Ready to move to PLAN phase

## Remember

**Design is where strategy happens.**

A good design:
- Aligns approach with intent
- Surfaces trade-offs explicitly
- Gets stakeholder buy-in
- Prevents tactical drift

**The Rule:** Never proceed to PLAN without approved design.
